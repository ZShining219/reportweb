# AGENTS.md

## 基本协作原则

本项目默认使用中文进行沟通、进度反馈和文档维护。

Agent 在执行任务前，应先理解用户需求、当前项目状态和 Git 状态，再进行修改。除非用户明确要求，不主动扩大任务范围，不修改与当前任务无关的文件。

遇到不确定、冲突或高风险操作时，应先向用户说明。

## 项目定位

本项目是 ReportWeb / ReportWebShow 的 Web 汇报演示与数据模型验证项目，目标是建立一套“Codex 内容整理 + 标准数据文档 + Web 呈现端 + 前端可编辑 revision”的工作流。

项目当前处于 Demo 基座阶段，应优先保持数据模型、前端呈现和 revision 机制清晰可验证，避免过早引入复杂框架或大范围重构。

当前核心能力包括：

- `server.js`：Node HTTP 服务，负责静态资源、汇报数据读取、revision 保存和当前 revision 切换。
- `src/`：汇报数据与 patch/revision 合并逻辑。
- `public/`：前端 Demo 呈现端，包含播放模式、编辑模式、节点导航和组件操作。
- `data/`：样例 sources、reports、patches 和 schema 说明。
- `assets/`：汇报中引用的 SVG 等静态资源。
- `docs/`：长期设计文档与数据/呈现模型说明。
- `tests/`：Node 内置测试，覆盖核心数据合并和节点图逻辑。

后续功能应围绕汇报生产闭环建设：来源记录、基础汇报生成、树形叙事、前端微调、revision 回撤、patch 固化和最终展示。

## 目录与职责边界

- 数据结构和样例内容优先放在 `data/`，并与 `data/schemas/` 或 `docs/design/` 中的结构说明保持一致。
- 服务端接口与读写逻辑优先放在 `server.js` 和 `src/`，保持无框架 Demo 的轻量边界；如需引入主要后端框架，应先获得用户确认。
- 前端展示、交互和样式放在 `public/`，应尽量复用现有 DOM + CSS 模式，不为局部功能随意引入构建链。
- 汇报素材、SVG、图片等静态资源放在 `assets/`，数据文件中只保存引用路径，不直接嵌入大段资源内容。
- 长期设计、流程和模型说明放入 `docs/`；临时分析和过程草稿不要散落在根目录。
- 测试应优先覆盖 patch 合并、revision 父链、节点导航、保存冲突等核心行为。

## Git 版本管理

Agent 在开始有实际影响的工作前，必须检查并反馈当前 Git 状态。

每次关键阶段反馈时，应使用以下格式：

```text
Git 状态：
- 当前分支：
- 本地变更：有/无
- 是否执行 commit：是/否
```

禁止覆盖、丢弃或回滚用户已有改动。禁止在未获得用户明确授权的情况下执行破坏性 Git 操作，包括 `git reset --hard`、`git checkout -- .`、强制推送等。

开发工作优先使用功能分支。不得默认直接 push 到 `main`。当用户要求 push 时，必须先询问目标分支；如果目标是 `main`，必须再次明确确认。

push 前必须反馈当前分支、目标远程分支、将推送的提交和变更文件。

当前目标远端仓库：

- GitHub 页面：`https://github.com/ZShining219/reportweb`
- Git SSH 连接路径：`git@github.com:ZShining219/reportweb.git`
- 默认远端名：`origin`

本项目 Git 远端连接默认走 SSH 密钥，不使用 HTTPS。检查与设置命令：

```shell
git remote -v
git remote set-url origin git@github.com:ZShining219/reportweb.git
ssh -T git@github.com
```

如果远端仓库尚未建立分支或当前本地分支尚未设置 upstream，只能在用户明确确认后执行首次 push。不得为了“连通性测试”自动 push。

## Codex 进度反馈

多步骤任务中应及时反馈进度，尤其是在开始修改前、完成主要修改后、完成验证后、准备 commit 或 push 前。

反馈应简洁明确。若没有产生更新，也需要说明“当前无新增变更”。

## UI 与交互风格管理

UI 修改应优先延续 `public/styles.css` 中的整体视觉语言、布局节奏和 CSS 变量。不要在局部交互中随意硬编码颜色、间距、字号、圆角、阴影等风格值。

如确需新增可复用视觉规则，应优先沉淀为 CSS 变量、公共 class 或 `docs/design/` 设计说明，再在具体组件中引用。涉及汇报画布、节点导航、编辑控件或 revision 面板的交互调整，应同步检查是否需要更新文档或测试。

## Codex 临时文档归档

Codex 在开发过程中产生的临时分析、方案草稿、过程记录、排查记录、讨论整理等，不得散落在项目根目录或业务目录。统一放入 `.local/codex/`，该目录仅作为本地临时工作区，不应提交。

临时文档命名建议使用半规范格式：`YYYYMMDD-主题.md`。只有用户明确要求沉淀为正式文档，或内容已经稳定到可作为项目长期参考时，才整理进入 `docs/`，并在 README 或相关索引中提供入口。

## README 维护

`README.md` 是项目的主要入口和导航文档，应保持清晰、可检索、可持续更新。

README 不要求承载所有细节，也不应被固定模板限制。随着项目发展，README 的结构可以调整、合并、拆分或重组，但应始终服务于：

- 快速理解项目是什么；
- 快速找到运行、开发、部署和文档入口；
- 保持当前项目状态、重要入口和关键链接最新；
- 避免与 `docs/` 中的详细文档长期重复。

当项目定位、运行方式、目录结构、开发流程、部署方式或重要功能发生变化时，应检查 README 是否需要同步更新。

复杂说明优先放入 `docs/`，README 负责提供清晰入口和索引。

## 验证要求

修改代码后，应根据改动范围进行必要验证。

常见验证方式：

- 文档或 Git 配置类改动：至少执行 `git status`、必要的文件内容检查或远端只读连通性检查。
- 数据与 patch 逻辑：运行 `npm test`，重点关注 revision 父链、patch 合并和冲突检测。
- 服务端接口：运行 `npm start` 后检查 `/`、`/api/reports` 和目标汇报 API 是否可访问。
- 前端交互：涉及 UI 或交互时，应在浏览器中检查播放/编辑模式、节点切换、保存 revision 和回撤切换等关键路径。
- 数据或文档结构变化：同步检查 README、`docs/` 和 `data/schemas/` 是否需要更新。

反馈时应明确说明：

```text
验证状态：
- 是否执行验证：是/否
- 验证命令：
- 验证结果：
```

如果未执行验证，需要明确说明“未执行验证”。

## 安全与敏感信息

不得提交密钥、token、账号密码、私有证书、`.env` 等敏感信息。

如需新增环境变量，应提供示例文件或文档说明，不直接写入真实敏感值。

## 高风险操作确认

以下操作必须先获得用户明确确认：

- push 到远程仓库；
- push 到 `main`；
- 删除文件或目录；
- 回滚、重置或覆盖已有改动；
- 引入新的主要依赖；
- 改变项目技术栈；
- 执行可能影响本地环境的命令。
